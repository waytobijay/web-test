🗺️ How It All Works Together

The Big Picture

Your website is made of three layers:

LayerWhat It IsWhere Changes Come From
data.json The content file — stores all text, products, settings, prices, theme colors You edit in admin panel → click "Download data.json" → upload to GitHub
localStorage / Firebase Order database — stores customer orders and enquiries Customers place orders → saved automatically (no download needed)
HTML / CSS / JS files The actual website code — you rarely need to touch these Developer changes only
📌
Important: When you make any content changes in the admin panel, those changes only go live after you click "⬇ Download data.json" (in the Theme tab) and commit that file to GitHub. Orders and enquiries save automatically — no download needed.
⚡ No Flash of Default Content

The website starts completely invisible (opacity 0) until your data.json has finished loading. Once all your content is ready, the page fades in — so visitors always see your latest content, never a blank default. This eliminates the flicker of placeholder text that would appear before your custom data loaded.

✅
This means every page load shows your real content from the first frame. There is a 2.5 second safety timeout — if data.json fails to load, the page reveals itself anyway so visitors are never stuck on a blank screen.

📋 Contact Form Fields — Add & Customise

What Is This?

The contact form your visitors fill in is now fully customisable from the admin panel. You can add new fields (phone, company name, budget, etc.), remove ones you don't need, mark them as mandatory or optional, and reorder them — all without touching any code.

📍 Contact Us form on the main website
Managing Existing Fields

Each field shows as a card in the Contact tab. For every field you can:

ControlWhat It Does

Mandatory checkboxWhen checked, the field gets a red asterisk (*) and the form will not submit unless the visitor fills it in
LabelThe text the visitor sees above the input box (e.g. "Your Name")
PlaceholderThe faded hint text inside the input box before they type (e.g. "Enter your full name")
↑ / ↓ arrowsMove the field up or down in the form order
SaveSaves your label/placeholder changes
DeletePermanently removes the field. Default fields (Name, Email, Message) should not be deleted
Default Fields (Out of the Box)
FieldTypeDefault State
Full NameText✅ Enabled, ✅ Mandatory
Email AddressEmail✅ Enabled, ✅ Mandatory
Phone NumberTel⛔ Disabled (enable to show)
SubjectText✅ Enabled, optional
MessageTextarea✅ Enabled, ✅ Mandatory
Adding a New Field
  1. Scroll to the bottom of the Contact Form Fields card → click "+ Add New Field"
  2. Enter the Label (e.g. "Company Name")
  3. Enter the Field Key — a short internal identifier, no spaces (e.g. company). This is used in the enquiry data
  4. Choose the Type: Text, Email, Tel, Number, Textarea, or Select (dropdown)
  5. Click "Add Field" — the field appears in the list enabled by default
  6. Click Save on the field card, then Download data.json and push to GitHub
💡
The field key (e.g. company) will appear in the Enquiries tab next to the submitted value. Keep it short, lowercase, and with no spaces — e.g. budget, phone, company_name.

📦 Orders Tab

What Is This Tab?

Your order management centre. Every order placed by a customer on the website appears here with full details: who ordered, what they ordered, how much, which payment method, and current status.

Order Status Pipeline

Every order moves through a status pipeline as your team processes it:

🟡 Order PlacedCustomer checked out
🟠 At WarehouseBeing processed
🔵 Ready for DeliveryPacked & ready
🟣 Out for DeliveryDriver dispatched
🟢 DeliveredConfirmed
StatusMeaningWho Updates It
🟡 Order PlacedCustomer just completed checkoutAutomatic
🟠 At WarehouseWarehouse team has received and is processing the orderWarehouse Admin (admin-warehouse.html)
🔵 Ready for DeliveryOrder is packed and waiting for pickup by delivery driverWarehouse Admin
🔴 Not ReadyItem is out of stock or has an issue — order cannot proceed yetWarehouse Admin
🟣 Out for DeliveryDriver has collected the order and is en route to customerDelivery Admin (admin-delivery.html)
🟢 DeliveredCustomer confirmed they received the orderQuality Admin (admin-quality.html)

⚫ CancelledOrder was cancelledMaster Admin
How to Use
  • Use the status filter dropdown to see orders in a specific stage
  • Click "View" on any order row to see full details and update its status
  • Add a comment when changing status (e.g. "Item out of stock — expected restock 5 Jan")
  • Click "⬇ Export Excel" to download all orders as a spreadsheet
  • Click "🔄 Refresh" to reload the latest orders from the database
📌
Storage note: By default, orders are stored in your browser's local storage. This means orders placed on one device are only visible from that same browser. To see all orders from all customers across all devices, set up Firebase in the Configuration tab (explained in detail below).

⚡ Test Google Connection Button

Clicking "Test Google Connection" in the admin panel checks that Firebase Authentication is reachable with your API key. A green result confirms the endpoint is working. If Google Sign-In fails during actual use, it means the Google provider isn't enabled in Firebase Console yet (follow setup steps below).

Setting Up Firebase Authentication (One-Time, ~5 minutes)

The Sign In button and Google button will be visible immediately. For sign-in to actually work, you need to enable providers in Firebase:

  1. Go to console.firebase.google.com → select your project
  2. Left sidebar → Authentication → Get started
  3. Sign-in method tab → click Email/Password → Enable → Save
  4. Click Google → Enable → add your Gmail as support email → Save
  5. Click Settings tab → Authorized domains → Add domain → enter your GitHub Pages domain (e.g. yourname.github.io)
  6. Back in admin panel → ⚡ Test Firebase Auth Connection to confirm it works
💡
If you have already set up Firebase for orders (data backend), authentication uses the same project and API key — no extra setup for that part.

📧 EmailJS Configuration

Required for two features: (1) the Contact Form sending messages to you, and (2) the Payment Confirmation Email notifying you of orders.

  1. Create a free account at emailjs.com
  2. Add an email service (Gmail, Outlook, etc.) → copy the Service ID
  3. Create an email template → copy the Template ID
  4. Go to Account → copy your Public Key
  5. Paste all three into the Configuration tab → click "Save EmailJS"
⚠️
Without EmailJS configured, the contact form will show an error and no confirmation emails will be sent. Orders still save normally — it only affects email notifications.

👤 User Profile Panel & Favourites

What Is the Profile Panel?

A full-screen side panel that slides in from the right when a signed-in customer clicks their name in the navigation bar. Designed like Amazon or Shopify's account area — modern, clean, and functional.

📍 Slides over the website from the right side
Profile Tab — Overview

The first tab customers see when they open their profile. Shows a summary of their activity:

  • Name & email — their account details
  • Stats grid — Total Orders, Favourites count, Member Since date
  • Recent orders — their last 3 orders with status badges

Orders Tab

Shows the full order history for the signed-in customer. Each row includes:

  • Current status badge (e.g. 🟡 Order Placed, 🟢 Delivered)

Order history requires Firebase to be connected. With Local Storage backend, orders are saved in the browser and may not match across devices.

Shows all products the customer has saved as a favourite. Customers add favourites by clicking the heart icon on any product card on the main website.

  • Clicking the heart again removes it from favourites

Lets customers update their account details:

🛒 Guest Checkout

What Is This?

When User Accounts is ON and Allow Guest Checkout is ON, customers who are not signed in will see a choice overlay when they click "Proceed to Checkout". This gives them two options:

OptionWhat Happens
Create an Account / Sign In Opens the auth modal. Once signed in, checkout proceeds and the order is linked to their account (visible in their Orders tab)
Continue as Guest Skips authentication entirely. The regular 3-step checkout proceeds as normal. The order is saved but not linked to any user account
💡
If Allow Guest Checkout is OFF and User Accounts is ON, customers must be signed in to complete a purchase. This is useful if you want all orders to be linked to accounts — but it reduces conversion for first-time buyers who don't want to register.
Recommended Setting

For most businesses:

  • Enable User Accounts so customers can register and track orders
  • Enable Allow Guest Checkout so you don't lose sales from visitors who just want to buy quickly
  • Enable Google Sign-In — it's the fastest way for customers to register (one click, no password)
  • Enable Email Verification only if you need to prevent fake accounts

🗄 Backend Options — Full Explanation

Why Does This Matter?

Your website needs to store orders and enquiries somewhere. The backend setting determines where that data lives and who can see it. This is the single most important technical decision for running the business side of your website.

Option 1 — 💾 Local Storage (Default)

What Happens

Data is saved inside the visitor's browser on their own device. Think of it like a notebook that only exists on that one computer.

ScenarioWhat Happens
Customer A orders from their phoneOrder saved in Customer A's phone browser
You open admin panel on your laptopYou see Customer A's order only if you use the same browser
Your warehouse manager opens adminThey see a different (their own) local storage — empty
You clear your browser cacheAll orders are permanently deleted
✅
Best for: Testing the system, learning how it works, single-admin operations where you want to track your own enquiries only.
❌
Not suitable for: Real customer orders that need to be shared with a warehouse team, or any production use where orders must not be lost.

Option 2 — 🔥 Firebase (Recommended)

What It Is

Firebase Firestore is a cloud database made by Google. All orders and enquiries are saved to Google's servers and can be viewed from any browser, any device, anywhere in the world — instantly.

What Happens When Firebase Is ON
ScenarioWith Firebase
Customer orders from their phone✅ Order saved to Firebase cloud immediately
You open admin on your laptop✅ You see the order within seconds
Warehouse manager opens admin-warehouse.html✅ They also see the same order
You close your browser✅ Order still exists safely in Firebase
1000 customers order in a day✅ All orders visible (free tier: 50K reads/day)
How to Set Up Firebase (15 minutes, free)
  1. Go to console.firebase.google.com and sign in with your Google account
  2. Click "Create a project" → give it a name (e.g. "bista-group") → Continue
  3. In the left sidebar, click Firestore Database → Create database
  4. Choose Start in test mode (you can add security rules later) → select a region close to Nepal (e.g. asia-south1) → Enable
  5. Click the ⚙ Settings gear → Project Settings → scroll down to "Your apps" → click Web (</>) icon
  6. Give the app a nickname → click Register app
  7. You will see a block of code with your configuration. Copy these values:
Config KeyWhere to Paste in Admin
apiKeyConfiguration tab → Firebase → API Key
projectIdConfiguration tab → Firebase → Project ID
authDomainConfiguration tab → Firebase → Auth Domain
storageBucketConfiguration tab → Firebase → Storage Bucket
appIdConfiguration tab → Firebase → App ID
  1. In admin panel → ⚙ Configuration tab → select Firebase radio button
  2. Paste all the values → click "Save Backend Config"
  3. Refresh the page → from now on, all orders save to Firebase
✅
Firebase Free Tier (Spark Plan) includes: 50,000 document reads/day, 20,000 writes/day, 1 GB storage. This is more than enough for a growing business. You only pay if you exceed these limits consistently.

Option 3 — ⚡ Supabase

What It Is

Supabase is an open-source alternative to Firebase, built on PostgreSQL (a powerful database). It offers similar cloud-sync features with a generous free tier.

How to Set Up Supabase
  1. Create a free account at supabase.com
  2. Create a new project → choose a region and set a database password
  3. Wait 2 minutes for the project to be created
  4. Go to Project Settings → API
  5. Copy the Project URL and anon/public key
  6. In admin → Configuration tab → select Supabase → paste both values → Save
  7. Create tables in Supabase: orders, enquiries, admins (with columns matching the order object structure)
⚠️
Supabase requires you to manually create database tables. Firebase is recommended for non-technical users as it requires no table setup — it creates everything automatically.

Summary: Which Backend Should You Choose?

Your SituationRecommended Backend
Just testing the system / learning💾 Local Storage (default)
Running a real business with real customers🔥 Firebase
You have a technical team and prefer open-source⚡ Supabase
Multiple staff need to see orders simultaneously🔥 Firebase
Small volume, single admin, just contact enquiries💾 Local Storage is fine

🔑 Role-Based Admin Portals

Overview

Instead of giving everyone the full admin panel (which could be overwhelming or risk accidental changes), you can create separate logins for each staff member with access to only what they need.

PortalURLWho Uses ItWhat They Can Do
🔑 Shared Loginadmin-login.htmlAll role adminsSelect their role, log in, get redirected to their portal
🏭 Warehouse Portaladmin-warehouse.htmlWarehouse/packing staffSee new orders, mark as At Warehouse / Ready / Not Ready with comments
🚚 Delivery Portaladmin-delivery.htmlDispatch / drivers teamSee ready-to-ship orders, mark as Out for Delivery or Not Delivered
✅ Quality Portaladmin-quality.htmlCustomer service / QCConfirm delivery, record customer rating (1–5 stars) and feedback notes
How an Order Flows Through the Team
  1. Customer places an order on the website → Status: 🟡 Order Placed
  2. Warehouse Admin sees it in their portal → processes the items → updates to 🟠 At Warehouse
  3. Once packed → Warehouse Admin marks it 🔵 Ready for Delivery
  4. Delivery Admin sees the ready order → dispatches driver → marks 🟣 Out for Delivery
  5. Quality Admin confirms delivery was received → marks 🟢 Delivered → records customer feedback
  6. Master Admin can see the full history of all status changes with timestamps and who made each update
📌
All portals show live stats at the top (e.g. "5 Ready for Delivery", "2 Not Delivered") so staff can quickly see their workload without scrolling through tables.

🚀 Publishing Changes to the Live Website

The Two Types of Changes
Type of ChangeExamplesHow to Publish
Content Changes Products, prices, text, images, theme colors, payment gateways, design Admin Panel → edit → Download data.json → replace file → push to GitHub
Order/Enquiry Data New orders from customers, enquiry submissions, status updates Automatic — saves directly to local storage or Firebase. No download needed
Step-by-Step: Publishing Content Changes
  1. Open admin panel (admin.html) — either locally or on GitHub Pages
  2. Make your changes in the relevant tab (e.g. update a product price)
  3. Click the Save button in that tab (e.g. "Save Product")
  4. Go to the Theme tab → click "⬇ Download data.json"
  5. A file called data.json will download to your computer
  6. Open your project folder → delete the old data.json → move the new one in
  7. Open GitHub Desktop (or terminal) → commit with a message like "Update product prices"
  8. Push/Sync to GitHub → your live website updates within 1–2 minutes
