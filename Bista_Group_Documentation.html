1. Full Project Overview

1.1 Purpose of the Project

The Bista Group website is a professional corporate web presence for a diversified conglomerate headquartered in Kathmandu, Nepal. It delivers a fully manageable public-facing website showcasing the company brand, products, leadership team, gallery, and contact information — without requiring developer involvement for content updates.

The website is built around a dual-panel architecture:

  • Public Website (index.html) — Visible to all internet visitors. Displays all company content dynamically, loaded in real time from the Firebase cloud database.
  • Admin Panel (admin.html) — A password-protected control panel for authorised personnel. Enables complete content management: text edits, image uploads, product management, team updates, gallery management, and theme customisation.

The system runs without a traditional server or backend framework. All logic runs in the visitor's browser using vanilla JavaScript, with three cloud services handling storage, images, and delivery.

1.2 End-to-End System Architecture

Cloud ServiceRole in the System
GitHub PagesHosts all HTML, CSS, and JavaScript files. Serves the public website and admin panel to any visitor worldwide for free with automatic HTTPS.
Firebase Realtime DatabaseCloud database. All text content, product data, team information, settings, and image URLs are stored here and read by every visitor on page load.
CloudinaryImage hosting and CDN. Images are uploaded directly from the browser to Cloudinary. A permanent https:// URL is returned and saved to Firebase.

1.3 Complete Data Flow Sequence

  1. Admin opens admin.html and logs in with the username and password.
  2. Admin selects a section tab and makes changes — editing text, uploading images, or managing products.
  3. For images: admin.js sends the file directly to the Cloudinary API via a browser fetch() POST. No server is involved.
  4. Cloudinary stores and optimises the image and returns a permanent https:// URL.
  5. admin.js writes all data (including the Cloudinary URL) to Firebase at path siteData/{sectionKey}.
  6. Firebase stores the data and makes it immediately available to all readers worldwide.
  7. Any visitor opening index.html triggers script.js, which reads the complete siteData object from Firebase.
  8. script.js renders all page sections with the live Firebase data.
  9. Changes are globally visible within seconds — no deployments, restarts, or file edits needed.

2. Detailed Section Breakdown

2.1 Public Website Sections

Hero siteData/hero

The first visual element visitors see. Contains three editable fields: tag (short promotional label), title (main heading), and sub (brand subtitle). Sets the tone and brand identity of the entire site.

Logo siteData/logo

Controls the logo image displayed in the navigation bar and footer, plus the brand name text. The logo is uploaded to Cloudinary and its URL stored in Firebase. The brand name also serves as the browser tab title.

Presents the company story and statistics. Contains: a company image (Cloudinary), a badge showing years of experience, two description paragraphs, and three statistic counters (e.g. 500+ Products, 50+ Countries, 10K+ Customers).

A responsive grid of product cards. Each card shows a name, short description, price in NPR with automatic conversion to AUD and USD, and an image. Clicking opens a modal with the full description. Full CRUD available in the admin panel.

Team member cards with a Cloudinary photo, name, role title, and biography. Full CRUD available in the admin panel.

Gallery siteData/gallery

A responsive image grid of company events and activities. Each item has a Cloudinary image URL and optional caption. Images can be added and removed from the admin panel.

Testimonials siteData/testimonials

Client reviews with name, company/title, review text, and a 1–5 star rating. Full CRUD available in the admin panel.

Contact siteData/contact

Company email, phone, and address. Includes a functional contact form powered by EmailJS — visitors can send messages with no server-side code.

Footer

Displays the brand name, logo, description, repeated contact details, and copyright notice. All content is pulled from the Firebase logo and contact data for consistency.

3. Tools and Services Explanation

3.1 Firebase Realtime Database

Purpose: Cloud-hosted NoSQL database from Google. Single source of truth for all website content — every text field, product, team member, gallery item, testimonial, and setting.

How it is used: Data is stored at siteData/{key}. The admin panel writes data via the Firebase JS SDK (loaded from CDN). The public site reads all data on every page load via a single database call.

Project ID    : bista-group
Database URL  : https://bista-group-default-rtdb.firebaseio.com
SDK version   : Firebase Compat v9.23.0 via CDN
Data path     : siteData/{sectionKey}
Rules mode    : test mode — .read = true, .write = true

Basic Usage:

  1. Go to console.firebase.google.com and open your project.
  2. Navigate to Build → Realtime Database.
  3. Use the Data tab to view all stored content in real time.
  4. Use the Rules tab to manage read/write permissions.

Troubleshooting:

  • Changes not visible to visitors: Firebase Rules have .read set to false. Set both .read and .write to true.
  • Login does nothing: firebase.database() crashed the script. Check databaseURL is correct and the Realtime Database has been created (not just the project).
  • Database not found error: Firebase Console → Realtime Database → Create Database → Select Region → Start in Test Mode → Enable.
  • Script crashes on load: Firebase init is now wrapped in try/catch — errors are caught and the page still loads.

3.2 Cloudinary

Purpose: Cloud media management. Permanently stores all uploaded images and delivers them via global CDN. Replaces the need for server-side file storage.

How it is used: When an admin uploads an image, admin.js sends the file to the Cloudinary Upload API using fetch() POST with FormData. No server required. Cloudinary returns a permanent https:// URL which is then saved to Firebase.

Cloud Name    : dyjzhbvkf
Upload Preset : bista-group-unassigned
Preset Type   : Unsigned (no server signing required)
Upload Folder : bista-group
API Endpoint  : https://api.cloudinary.com/v1_1/dyjzhbvkf/image/upload

Troubleshooting:

  • Upload fails silently: Verify the preset name in admin.js matches exactly — names are case-sensitive.
  • Unauthorised error: Preset Signing Mode must be Unsigned. Signed presets require server-side API keys.
  • Image disappears on reload: The URL was not saved to Firebase. Always click the Save button after uploading.

3.3 GitHub Pages

Purpose: Free static site hosting by GitHub. Serves all HTML, CSS, JS, and assets from a public repository with automatic HTTPS at zero cost.

Basic Usage:

  1. Create a public repository on GitHub.
  2. Upload all project files to the repository root.
  3. Go to Settings → Pages → Branch: main → Folder: / (root) → Save.
  4. Wait ~60 seconds. Site is live at https://USERNAME.github.io/REPO-NAME/.
  5. To update: simply upload changed files to the repository.

Troubleshooting:

  • Site not loading: Repository must be Public.
  • Old content showing: Hard refresh: Ctrl+Shift+R (Windows) or Cmd+Shift+R (Mac).
  • Images broken: GitHub Pages runs on Linux — file paths are case-sensitive.

3.4 EmailJS

Purpose: Sends emails from client-side JavaScript with no server required. Powers the contact form on the website.

Required template variables: from_name, reply_to, subject, message

Setup: Sign up at emailjs.com → connect Gmail → create template → copy Public Key, Service ID, and Template ID → enter all three in Admin Panel → EmailJS Config tab.

Troubleshooting:

  • Form fails: Verify all three keys are saved in Admin Panel.
  • No email received: Check EmailJS dashboard for failed sends.
  • Empty variables in email: Use exact variable names listed above.

3.5 Python Development Server (server.py)

Purpose: Local HTTP server that serves files AND handles image uploads. Required because browsers block fetch() requests when pages are opened using the file:// protocol.

Usage:

  1. Open a terminal in the project folder.
  2. Run: python server.py
  3. Open browser at http://localhost:8080

Troubleshooting: If python is not found, try python3 server.py. If port is in use, change PORT = 8080 in server.py.

4. Process Workflow

4.1 Initial Setup — Six Phases

Phase 1: Create Service Accounts

  1. GitHub: Create an account at github.com or log in.
  2. Cloudinary: Create a free account at cloudinary.com.
  3. Firebase: Create an account at console.firebase.google.com using a Google account.
  4. EmailJS (optional): Create an account at emailjs.com if the contact form is required.

Phase 2: Configure Firebase

  1. In Firebase Console, click Add Project and enter the project name.
  2. Disable Google Analytics if not needed. Click Create Project.
  3. Navigate to Build → Realtime Database in the left sidebar.
  4. Click Create Database → Select a region → Start in Test Mode → Enable.
  5. Go to Project Settings (gear icon) → General → Your Apps.
  6. Click the web icon (</>) → Register app → Copy the firebaseConfig object.

Phase 3: Configure Cloudinary

  1. Log in to cloudinary.com → Settings → Upload tab.
  2. Scroll to Upload Presets → Add Upload Preset.
  3. Enter a preset name → Set Signing Mode to Unsigned → Set Upload Folder.
  4. Click Save and note the exact preset name.

Phase 4: Configure the Code

  1. Open js/admin.js — find the FIREBASE INIT block — paste your Firebase credentials.
  2. Set CLOUDINARY_CLOUD and CLOUDINARY_PRESET in the same file.
  3. Open js/script.js — paste the same Firebase credentials in its FIREBASE INIT block.
  4. Update ADMIN_USER and ADMIN_PASS in admin.js with secure credentials.

Phase 5: Deploy to GitHub Pages

  1. Create a new public repository on GitHub.
  2. Upload all project files to the repository root.
  3. Go to Settings → Pages → Branch: main → Folder: root → Save.
  4. After ~60 seconds, the site is live at the displayed URL.

Phase 6: Verify Everything Works

  1. Open the live site URL — verify default content loads.
  2. Open admin.html — log in with your credentials.
  3. Edit a text field → click Save → open index.html in a new tab → verify the change appears.
  4. Upload an image → click Save → verify it displays from a Cloudinary https:// URL.

4.2 Daily Content Update Workflow

  1. Navigate to your-site.github.io/admin.html
  2. Enter username and password → click Login.
  3. Click the tab for the section you want to update.
  4. Make changes: edit text, upload images, add/edit/delete items.
  5. Click the Save button. Green toast confirms success.
  6. Open the main website in a new tab to verify the change is live.

5.2 Updating Required Fields for a New Project

1 — Firebase Credentials (update in BOTH js/admin.js and js/script.js)

firebase.initializeApp({
  apiKey:            'YOUR_NEW_API_KEY',
  authDomain:        'YOUR_PROJECT.firebaseapp.com',
  databaseURL:       'https://YOUR_PROJECT-default-rtdb.firebaseio.com',
  projectId:         'YOUR_PROJECT',
  storageBucket:     'YOUR_PROJECT.firebasestorage.app',
  messagingSenderId: 'YOUR_SENDER_ID',
  appId:             'YOUR_APP_ID'
});

2 — Cloudinary Credentials (js/admin.js only)

const CLOUDINARY_CLOUD  = 'your_cloud_name';
const CLOUDINARY_PRESET = 'your_unsigned_preset_name';

3 — DEFAULTS Object (both JS files)

Update the DEFAULTS constant in both files with the new client content:

  • hero.title — Company name
  • hero.tag — Short promotional tagline
  • hero.sub — Brand description subtitle
  • logo.brand — Brand name for navigation and footer
  • contact.email / phone / addr — Contact details
  • products[] — Replace with real products (id, name, desc, fullDesc, price, img)
  • leadership[] — Replace with real team members (id, name, role, desc, img)
  • theme.primary / secondary / bg / accent — Brand colours in hex

4 — Admin Credentials (js/admin.js only)

const ADMIN_USER = 'your_admin_username';
const ADMIN_PASS = 'your_secure_password';
⚠ These are stored as plain text. Use a strong unique password and change before going live.

5.3 Modifying Website Sections

To add a completely new section:

  1. Add HTML markup in index.html with appropriate element IDs.
  2. Add admin form fields in admin.html within a new tab section.
  3. Add the default data structure to DEFAULTS in both JS files.
  4. Write a renderNewSection() function in script.js.
  5. Add a saveNewSection() function in admin.js.
  6. Call renderNewSection() inside renderAll() in script.js.
  7. Add population logic inside populateForms() in admin.js.

8. Reusable Skill — 10-Step Build Process

8.1 Required Accounts & Tools

  • GitHub account (github.com) — free static hosting
  • Firebase account (console.firebase.google.com) — cloud database
  • Cloudinary account (cloudinary.com) — permanent image storage
  • EmailJS account (emailjs.com) — optional, for contact forms
  • Text editor such as VS Code for configuration changes
  • Modern browser (Chrome recommended) for testing
  • Python — optional, for running the local development server

8.2 The 10-Step Repeatable Build Process

  1. Complete the Project Intake Form (Section 7) with the client before writing any code.
  2. Create Firebase project → Realtime Database → Test mode → Copy firebaseConfig.
  3. Create Cloudinary account → New unsigned upload preset → Note cloud name and preset name.
  4. Copy all template files from the Bista Group project to a new folder.
  5. Update DEFAULTS in both JS files with the new client brand content.
  6. Update FIREBASE_CONFIG in both JS files with new Firebase credentials.
  7. Update CLOUDINARY_CLOUD and CLOUDINARY_PRESET in admin.js.
  8. Update ADMIN_USER and ADMIN_PASS in admin.js with strong credentials.
  9. Replace logo and favicon in assets/images/ with client brand assets.
  10. Deploy — Create public GitHub repo → upload all files → enable GitHub Pages → verify live URL.

8.3 Post-Launch Verification Checklist

  • ☐   Main site loads at the GitHub Pages URL
  • ☐   Admin panel login works with new credentials
  • ☐   Text changes in admin panel appear on public site within seconds
  • ☐   Images upload to Cloudinary and display with permanent https:// URL
  • ☐   Firebase console shows siteData entries being written
  • ☐   Contact form sends email (if EmailJS configured)
  • ☐   Site displays correctly on mobile devices
  • ☐   All sections render with correct client content
  • ☐   Admin credentials changed from default (admin / bista2025)
  • ☐   Firebase Rules reviewed for production security
